# WebServer & ServerManager

This module provides a robust, generic foundation for connecting Unity Netcode clients (like VR headsets) to a Unity Server, while simultaneously discovering and communicating with an external Web Server running on the same local network.

At the core of this module is the `ServerManager.prefab`, which handles network bootstrapping, mDNS-based web server discovery, and connection configuration.

Extracted from our Centroid project. Python server and backend available at [ARTSLab-CITH/centroid-server](https://github.com/ARTSLab-CITH/centroid-server)

## Core Architecture

### The Tripartite Connection
The system is designed to support a three-way architecture:
1. **The Web Server (External)**: A custom web server (e.g., Python FastAPI) running on the local network, providing REST and WebSocket endpoints.
2. **The Unity Server (Local)**: A standalone Unity build or Editor instance running Netcode for GameObjects as a server. In this paradigm, it is assumed to be running on the *same machine* as the External Web Server.
3. **The Unity Clients (Headsets)**: Remote devices (like Android VR headsets) running the Unity application as Netcode clients. 

### ServerManager Core Components
The `ServerManager.prefab` combines standard Unity Netcode components with custom discovery logic:

*   **`ServerDiscovery.cs`**: Automatically locates the local Web Server using mDNS/Zeroconf.
    *   On Windows/Editor/Linux, it polls via Zeroconf.
    *   On Android, it uses the native Android NSD manager.
    *   Features exposed inspector properties (`serviceType`, `fallbackIp`, `fallbackPort`) making it framework-agnostic. 
*   **`NetworkBootstrap.cs`**: Automates connection flows. 
    *   **Server Build**: If compiled with `IS_SERVER` (or running as a server), it immediately starts the Netcode server on `0.0.0.0` to listen for incoming connections.
    *   **Client Build**: Waits for `ServerDiscovery` to successfully find the Web Server's IP address. Since the Unity Server and Web Server run on the same physical machine, it injects this discovered IP into the `UnityTransport` component and calls `StartClient()`.
*   **`AndroidMulticastLock.cs`**: A necessary utility script for Android builds (e.g., Quest headsets). It briefly acquires the Android OS Multicast Lock to permit UDP mDNS broadcast packets, allowing `ServerDiscovery` to function on constrained mobile devices.

## Example Usage (Centroid)

Contained in the `Example/` folder are scripts demonstrating how to extend and consume the core `ServerManager` to integrate with a specific external service (in this case, "Centroid").

These scripts should not be placed on the core `ServerManager.prefab` but instead added to scene objects depending on your logic requirements.

### 1. `CentroidClient.cs` (REST API)
An example of wrapping HTTP requests to the discovered web server. It waits for `ServerDiscovery.IsServerFound` to become true, then reads `ServerDiscovery.ServerUrl` to construct full endpoint requests (`POST /centroid/transcribe`). It is designed to be purely a client of the web server.

### 2. `CentroidSubscriber.cs` (WebSocket API)
An example of maintaining streaming state. It is restricted to run **only** on the Unity Server instance. 
Once `NetworkBootstrap` has determined the Netcode role, and `ServerDiscovery` has located the external web server, this script opens a persistent WebSocket (`WS /centroid/subscribe/...`). 
Because the Unity server acts as the central hub, it listens to the web server's event stream ("emotion_state", "centroid_response").

### 3. `CentroidNetworkManager.cs` (Netcode Sync)
An example of syncing state from the Unity Server down to the Unity Clients. 
When the Unity Server receives a WebSocket event through `CentroidSubscriber`, those events are translated into Netcode RPCs (`[Rpc(SendTo.NotServer)]`) by this `NetworkBehaviour`. Headsets subscribing to its C# events receive real-time updates originally generated by the remote Web Server without needing to connect strictly to the external Web Server themselves.
